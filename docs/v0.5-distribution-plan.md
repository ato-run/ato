## 0. Plan Identity — why "Plan B"

Earlier draft (Plan A) required corporate Apple Developer Program registration
(D-U-N-S + $99/yr + multi-week lead time) for macOS Desktop notarization.

Plan B replaces Apple-notarized signing with **ad-hoc signing
(`codesign --sign -`) + Homebrew Cask + first-class source-build path**, which:

- Reduces release cost to **$0**
- Cuts critical path from **W6 → W3** (no D-U-N-S wait)
- Keeps PR-3 dual-mode so a future Developer ID upgrade is an env var flip
- Matches the developer audience of a CLI-first product

The structural conclusion that **PWA cannot replace ato-desktop** (process model,
custom protocol per pane, secret store, subprocess sandbox all require a real
host process) holds regardless of plan choice — see §7-3.

## 3. Critical Path

### 3.1 (Removed in Plan B — was Apple Developer Program lead time)

Plan B removes Apple Developer Program registration from v0.5 critical path.
A future Developer ID upgrade is tracked as a v0.6+ candidate (see §7-1).

### 3.2 cargo-dist scope boundary

cargo-dist generates: CLI archive + Homebrew Formula + optional CLI MSI + sha256.

Out of cargo-dist scope (xtask + custom workflow): `.app` bundle, `.dmg`, Desktop `.msi`, `.AppImage`. These are uploaded to the cargo-dist–created GitHub Release as additional assets via `gh release upload`.

### 3.3 First-launch UX guidance

Because v0.5 macOS bundles are ad-hoc signed (not Apple-notarized), users
launching the `.app` outside Homebrew Cask will see "developer cannot be
verified". Mitigations:

- Homebrew Cask path: `auto_updates true` + post-install `xattr -dr com.apple.quarantine`
  removes the quarantine flag entirely (no warning).
- `install.sh` direct-download path: same `xattr -dr` after staging into `/Applications/`.
- README and release notes document the right-click → Open fallback.

### 7-1. v0.6+ Apple Developer ID upgrade
Trigger conditions (any one): GitHub stars ≥ 100, paying customers, security-sensitive deployment requests, brand perception complaints.
Migration cost: env var flip in PR-3 + GitHub Actions secret population. No code changes.

### 7-2. PoC cargo-dist Desktop bundle support
Time-boxed 1 day during W1 to confirm cargo-dist cannot replace xtask for `.app` / `.dmg` / `.msi` Desktop bundles. If it can (unlikely), PR-3 / PR-7 collapse into config.

### 7-3. PWA evaluation outcome — does NOT replace ato-desktop

Researched 2026-04-25. Conclusion: **structurally impossible**.

ato-desktop's defining responsibilities (PAS §3.2) cannot run in a browser:
- Subprocess spawn/kill of ato-cli (Wall 1)
- nacelle sandbox enforcement on capsule processes (Wall 2)
- Per-pane custom protocol `capsule<partitionId>://` cross-pane isolation (Wall 3)
- `~/.ato/secrets.json` plaintext-free local storage
- Deno/Node/Python runtime spawning
- `ato://` `capsule://` URL handler registration with the OS

A PWA would be a different product (browser-side capsule launcher with weaker
security model) and is **not** on the v0.5 / v0.6 roadmap. Document only;
no further planning effort allocated.

### 7-4. ato-cli `native_delivery` for ato-desktop self-distribution — confirmed NOT a v0.5 channel

Researched 2026-04-25 (deep-read of `apps/ato-cli/src/application/engine/build/native_delivery/` ≈ 4,000 LoC + `install/mod.rs`). Verdict: **NO at v0.5; possible v0.6+ side channel after extensions.**

Why native_delivery cannot replace the v0.5 SSOT (xtask + cargo-dist + Cask + MSI + AppImage):

1. **Different purpose** — native_delivery is *source-derivation install* (finalize on consumer host re-signs), not prebuilt-installer distribution. Every install runs `finalize_fetched_artifact` (`install/mod.rs:1377`).
2. **Linux finalize blocked** — `host_supports_finalize` (`filesystem.rs:639`) is `macos || windows` only. Linux end-users cannot complete a native install.
3. **`.dmg` / `.msi` / `.AppImage` not recognized** — `ensure_native_artifact_kind_supported` (`mod.rs:2373`) accepts only `.app` directories and `.exe` files. Single-file installers are rejected.
4. **No inside-out signing** — `FinalizeRunner` invokes `codesign` once on the outer bundle; cannot sign `Helpers/ato` first then `MacOS/ato-desktop` (PR-3 inside-out requirement).
5. **No URL scheme / PATH / `Program Files\Ato\` placement** — projection layer creates only a launcher symlink (`~/Applications/<Name>.app` mac/win, `~/.local/share/applications/*.desktop` + `~/.local/bin/` linux). No `lsregister`, no Windows registry writes for `URL Protocol`, no `xdg-mime default`, no PATH mutation.
6. **Bootstrap chicken-and-egg** — `ato install desky` requires `ato` already on PATH. Cannot deliver the *first* CLI install; install.sh / Homebrew / cargo-dist must remain primary.

What's already in place (useful later):
- Curated alias `desky → ato/desky` wired at `apps/ato-cli/src/cli/dispatch/install.rs:10`
- `.app` directory + `.exe` file are first-class artifact kinds
- Capsule supply-chain signature verification reused (`verify_manifest_supply_chain`)
- payload + derived integrity hash checks already cross-verified

To make native_delivery a viable v0.6+ side channel for ato-desktop, six extensions are needed (Linux finalize gate lift, multi-format input recognition, inside-out signing mode, post-install shell-integration hooks, `Helpers/ato` PATH exposure, installer-format wrappers). All are RFC-scale; tracked separately, not in v0.5.
