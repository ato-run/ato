これまでの深い議論とアーキテクチャ設計（ADR）を経て進化を遂げた **Ato（アト）** の全体像を総括します。

Atoを一言で表すなら、**「AIエージェント時代のための、アプリケーション志向なNix代替（Agentic, application-oriented Nix）」** です。

Nixの「完璧な再現性」、pnpmの「高速な依存共有」、Kubernetesの「インフラ抽象化」、そしてDockerの「隔離性」を統合しつつ、それらを **「開発者が独自言語（Nix式やDockerfile）を一切書かずに、ソースコードから自律的に推論・構築する」** という魔法のような体験（Agentic UX）として提供する次世代のメタランタイムです。

Atoのコアアーキテクチャは、以下の4つの柱で構成されています。

---

### 1. 究極のインフラ抽象化：5層モデルの `ato.lock.json`
Atoは、人間が手書きする曖昧な `capsule.toml` を廃止し、ソースコードや既存のエコシステム（`package.json` や `.env`）から推論された **`ato.lock.json` を単一の真実の情報源（SSOT: Canonical Input）** とします。

このロックファイルは、パッケージマネージャーとオーケストレーターの歴史的課題を解決する **5つの論理層** に分離されています。

1. **Resolution（再現性の核）**: ターゲットOS/Archごとのランタイム、依存関係（Closure）、環境要件を **Digest（ハッシュ）** として完全に固定します。
2. **Contract（ポータブルな要求）**: アプリが動くために「何が必要か（What）」を宣言します。（例：ポート3000、OIDC権限、メモリ2GB、Read-Onlyなファイルシステム）。
3. **Binding（環境固有の割り当て）**: Contractの要求を「今の環境でどう提供するか（How）」を定義します。（例：ホストのポート43123にマップする、ローカルの `~/.cache` をマウントする）。
4. **Policy（組織のガードレール）**: 「メモリ4GB以上は禁止」「特定の通信先以外はDeny」など、Contractの要求を制限・拒否（Fail-Closed）する関所です。
5. **Attestations（ミュータブルな学習）**: 実行時に未知の通信などが発生した際、プロンプトで承認された権限をここに記録します。ハッシュ（Resolution）を壊さずに学習が可能です。

### 2. 決定論的な砂時計型パイプライン（State Machine）
実行プロセスは、消費者（Run）と生産者（Publish）で共通化されたフェーズ（`Install -> Prepare -> Build -> Verify -> Dry-run -> Execute/Publish`）を厳格に遷移します。

* **バリア同期とDAG制御**: 複数のサービスを立ち上げる際、全サービスが `Verify`（検証）と `Dry-run`（事前シミュレーション）を完了するまで「バリア同期」で待機させます。安全が証明された後にのみ、依存関係（DAG）順に起動し、Readiness Probeでライフサイクルを管理します。
* **CleanupJournal によるフェイルクローズド**: 途中で失敗した場合、ジャーナルに記録された補償アクションを逆順に実行し、一時ディレクトリや子プロセスを跡形もなく消去して初期状態にロールバックします。

### 3. エージェント的な自律修復（Agentic Auto-fix & Inference）
未知のOSSリポジトリであっても、Atoは自律的に実行まで持っていく力を持っています。

* **自動推論**: `package.json` の `scripts.start` や、Vite/Next.jsなどの定番フレームワークから、コマンドや必要なポートを固定ルールで推論し `Contract` に書き込みます。
* **`--auto-fix` による修復**: 固定ポートが競合してSmokeテストが落ちた場合、Ato専用ポートプール（18000番台など）から空きポートを動的に探索し、`Binding` を書き換えて自動再試行します。
* **Nacelleサンドボックスでの依存解決**: ロックファイル（`package-lock.json` 等）が不足している場合、修復専用のサンドボックスを立ち上げて `npm install` 等を安全に実行し、依存関係のDigestを確定させます。

### 4. コマンドUXの分離（Immutable実行 vs Mutable開発）
「本番実行」と「ローカル開発」のUXをコマンドレベルで明確に分離し、Nixと同等の再現性をワンコマンドで提供します。

* **`ato run [source]` (不変な本番実行)**:
  アーティファクトやGitHubのソースを一時ディレクトリに展開し、Read-Onlyで安全に実行します。ホスト環境を一切汚さない、使い捨てのZero-Trust実行です。
* **`ato init [source]` (可変な開発環境の構築)**:
  ソースコードから開発環境を初期化します。現在のホストディレクトリをRead-Writeでバインドマウントするローカル用の `Binding` を生成し、必要なランタイム（Node, Python等）を瞬時にCASから引き当てます。開発者はNixを学ばずとも、隔離されたクリーンな開発環境（`ato shell` や `ato dev` へ繋がる土台）を即座に手に入れられます。

---

### 総括
Atoは、**「ソースコードを入力すると、推論と契約（Contract）に基づいて必要なランタイムと依存関係をかき集め、Zero-Trustなサンドボックス上で自律的かつ安全にオーケストレーションする基盤」** です。

ローカル開発の「自分の環境では動く」問題を撲滅するだけでなく、将来的にAIエージェントがインターネット上のソフトウェアを安全に調達・実行するための「標準プロトコル（Magnetic Webの実行基盤）」として機能するよう設計されています。
